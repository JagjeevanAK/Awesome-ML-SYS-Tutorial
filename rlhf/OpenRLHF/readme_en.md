## Computational flow of RLHF

It has been explained very clearly in [Illustrated Large Model RLHF Series: PPO Principles and Source Code Interpretation Everyone Can Understand](https://zhuanlan.zhihu.com/p/677607581). I will excerpt it here and add some more.

### Construct Reward

Given a transformer and any string, I can input the entire string to the reward model and do a forward pass to get the logit of the token at each position. We take out the logit of the last token, process it with the logit processor, perform softmax again and take the log, and get the log prob here. In addition, we can also perform other operations on the logit of the last token, such as pooling and projection, etc., to get embedding, reward or value. It can be seen that for each token in the string, we can get all the aforementioned calculated values. However, in RLHF, we will use the log prob and value of each token in the response, but the reward will only use the reward of the last token. The actual calculation of reward is given directly here:

$$
R_t =
\begin{cases}
-kl\_ctl \cdot \left( \log \frac{P(A_t|S_t)}{P_{ref}(A_t|S_t)} \right), & t \neq T \\
-kl\_ctl \cdot \left( \log \frac{P(A_t|S_t)}{P_{ref}(A_t|S_t)} \right) + R_t, & t = T
\end{cases}
$$

For the t-th response token, when t is the last token T, the reward for the entire response output by the reward model is added to $R_t$. In other words, in fact, a prompt + response will only cause the reward model to infer once, which will be used as the reward of the entire response.

As for other parts, $kl \_ ctl$ is a constant, $ \log \frac{P(A_t|S_t)}{P_{ref}(A_t|S_t)} $ is the logarithm of the conditional probability ratio of the token $A_t$ generated by the reference model and actor model, that is, directly subtracting the actor's log prob and the reference's log prob, which is reflected in the code as `kl_ctl * (actor_log_probs - ref_log_probs)` (KL divergence), thus obtaining the reward of each token. Note that the singular and plural here, `actor_log_probs` and `ref_log_probs` are lists composed of the log probs of all response tokens.

After obtaining the KL divergence, add the reward here (called reward score) to the last token of this `prompt + response`, and the reward for each part of the entire response is completed. Of course, the actual calculation also needs to consider the clipping problem of the reward score, that is, the reward cannot be made too large. There is a very good pseudo code in [Zhihu](https://zhuanlan.zhihu.com/p/677607581). Here is a code and analysis:

<details>
<summary>Pseudocode analysis</summary>

```python
def compute_rewards(self, prompts, log_probs, ref_log_probs, reward_score,
                        action_mask):
        """
        reward_function: Calculate the final reward score
        Review the default values of several related parameters:
        self.kl_ctl = 0.1
        self.clip_reward_value = 5
        
        For a certain prompt in the batch, its final reward score is:
        (1) First calculate the logit similarity between actor and ref_model: -self.kl_ctl * (log_probs - ref_log_probs)
            In fact, it is easier to understand if it is written as self.kl_ctl * (ref_log_probs - log_probs)
            The larger this value is, the higher the ref_model’s recognition of the results generated by the actor (that is, it indicates that rlhf has not been trained incorrectly).
            Even if there is no training error, we should also give some rewards to the model. This reward is self.kl_ctl * (ref_log_probs - log_probs)
            
        (2) Since we only take the score corresponding to the last token as reward_score, we only need:
            self.kl_ctl * (ref_log_probs - log_probs) last digit + reward_score
         
         (3) At the same time, we have also set a size limit on reward_score. The maximum size does not exceed self.clip_reward_value (any excess will be uniformly given as self.clip_reward_value).
             The minimum is not lower than -self.clip_reward_value (lower than is uniformly given as -self.clip_reward_value)
        
         (4) The size of the rewards finally returned is: (batch_size, the length of each piece of data). For each piece of data in the batch:
             - The last digit of response: self.kl_ctl * (ref_log_probs - log_probs) last digit + reward_score
             - The rest of the response: self.kl_ctl * (ref_log_probs - log_probs)
        
        """

        kl_divergence_estimate = -self.kl_ctl * (log_probs - ref_log_probs)
        rewards = kl_divergence_estimate
        # ------------------------------------------------------------------------------------------------------------------
        # The starting position of response
        # (Because we have done padding on the prompts, the length of each prompt in the batch is the same, which means that the starting position of each response is the same)
        # (So start here does not add s, just an int)
        # ------------------------------------------------------------------------------------------------------------------
        start = prompts.shape[1] - 1
        # ------------------------------------------------------------------------------------------------------------------
        # The end position of response
        # (Because the length of each response in a batch is different, the end position of the response is also different)
        # (So here end is added with s, and the size of ends is (batch_size,)
        # ------------------------------------------------------------------------------------------------------------------
        ends = start + action_mask[:, start:].sum(1) + 1
        # ------------------------------------------------------------------------------------------------------------------
        # Limit rewards_score
        # ------------------------------------------------------------------------------------------------------------------
        reward_clip = torch.clamp(reward_score, -self.clip_reward_value,
                                  self.clip_reward_value)
        batch_size = log_probs.shape[0]
        for j in range(batch_size):
            rewards[j, start:ends[j]][-1] += reward_clip[j] #

        return rewards
```
Pay attention to the dimensions of input and output. `prompts` is a matrix of `[batch size, padded prompt length]`. `ref_log_probs` and `log_probs` are matrices of `[batch size, padded prompt with response length]`. Then only the `reward` from `prompt` to `response` will actually have an effect. `prompt` `reward` is not calculated.

`prompt` has uniform `padding`, so the `start` position of `response` is unique, and `ends` is calculated from the 1-element cutoff in `action_mask`. Finally, in this `batch`, the `token` at the end of the `reward` of each `prompt` is added to the `reward_score` to get the `reward` obtained through clip.

### Construct Advantage

Advanatage can be understood as "unexpected surprise" to a certain extent. For the specific description, please refer to the original Zhihu article. The construction formula of Advantage is given directly here:

$$
Adv_t = \left( R_t + \gamma \cdot V_{t+1} - V_t \right) + \gamma \cdot \lambda \cdot Adv_{t+1}
$$

Let's break it down, considering that $R_t$ is the reward of each token, which has been constructed before. $V_t$ and $V_{t+1}$ are the values ​​of the current token and the next token, and the value of each token can be obtained in the forward pass of the value model. $Adv_t$ is the advantage of the current token, and $\gamma and \lambda$ are constants. This recursive construction method can use tail recursion to infer the advantage of each position.

Here is the pseudo code. Note that this function also returns `returns`, which is the actual income of each token. This income will be used to update the critic model later:

<details>
<summary>Pseudocode</summary>

```python
 def get_advantages_and_returns(self, values, rewards, start):
        """
        Adopted from https://github.com/CarperAI/trlx/blob/main/trlx/models/modeling_ppo.py#L134
        
        The advantage value at time t before GAE is introduced:
        detal_t = r_t + gamma * V_t+1 - V_t
        Among them:
            - r_t represents the immediate income at time t
            - V_t+1 represents the expected return in the future
            - r_t + gamma * V_t+1 can be understood as the actual expected return at time t
            - V_t can be understood as the estimated expected return at time t (estimated by the model, such as the critic model itself)
        
        The advantage value at time t after the introduction of GAE:
        A_t = delta_t + gamma * lambda * A_t+1
        A rough understanding means that at time t, not only the current advantages are considered, but also the future advantages are considered.
        In order to know A_t, we need to know A_t+1, so in this algorithm we adopt the method of dynamic programming from back to front, that is:
        Assuming T is the last time, then A_T+1 = 0, so: A_T = delta_T
        Once you know A_T, you can work backwards and calculate A_t-1, A_t-2, etc.
        
        The actual expected return at time t after the introduction of GAE
        returns_t = A_t + V_t
                  = delta_t + gamma * lambda * A_t+1 + V_t
                  = r_t + gamma * V_t+1 - V_t + gamma * lambda * A_t+1 + V_t
                  = r_t + gamma * (V_t+1 + lambda * A_t+1)
        
        Note that whether it is advantages or returns, they only count the response part.
        """
        
        # Adopted from https://github.com/CarperAI/trlx/blob/main/trlx/models/modeling_ppo.py#L134
        lastgaelam = 0
        advantages_reversed = []
        length = rewards.size()[-1]
        # Note that reversed is used here, which is a backward calculation method.
        for t in reversed(range(start, length)):
            nextvalues = values[:, t + 1] if t < length - 1 else 0.0
            delta = rewards[:, t] + self.gamma * nextvalues - values[:, t]
            lastgaelam = delta + self.gamma * self.lam * lastgaelam
            advantages_reversed.append(lastgaelam)
        advantages = torch.stack(advantages_reversed[::-1], dim=1) # Advantages
        returns = advantages + values[:, start:] # Actual benefits
        # values: expected return
        return advantages.detach(), returns
```

### Construct Actor Loss

Here is the construction formula of Actor Loss:

$$
actor \_ loss = - \min \left[ Adv_t \cdot \frac{P(A_t|S_t)}{P_{old}(A_t|S_t)}, Adv_t \cdot clip \left( \frac{P(A_t|S_t)}{P_{old}(A_t|S_t)}, 0.8, 1.2 \right) \right]
$$

This construction formula looks complicated, but in fact it is not simple at all. The construction of $Adv_t$ for each response token has been given above, and $P(A_t|S_t), P_{\text{old}}(A_t|S_t)$ are actually the conditional probabilities of the actor model. The reason there is old is because we want to make more use of the experiences generated in each round, so a set of experiences will be updated for multiple rounds. old means that this set of experiences is used to update the previous actor model, and this old actor model is used to constrain the size of these rounds of updates. **Finally, considering that in a certain round of updates, the gap between the current actor model and the old actor model is so large that the ratio of conditional probabilities exceeds the artificially preset range, at this time the coefficient (ratio) of $Adv_t$ will be taken as the constraint boundary. At this time, the parameters of the actor model no longer affect the ratio. In other words, the parameters of the actor model are no longer in the calculation graph of actor loss, and this loss will not update the parameters of the actor. ** Note that the advantage structure is constructed from the old actor model and is fixed after the calculation. There is no gradient for the updated actor model, so in the entire actor loss calculation graph, only ratio has a gradient for the updated actor model.

Here is the code. Note that the loss in each part of the entire response is averaged, which is the actor loss of prompt + response.

<details>
<summary>Pseudocode</summary>

```python

    def actor_loss_fn(self, logprobs, old_logprobs, advantages, mask):
        """
        logprobs: calculated in real time, the prob of the response part (only this changes with the real-time update of the actor)
        old_logprobs: the prob of the response part in the old strategy (this is fixed and does not change with the real-time update of the actor)
        Advantages: In the old strategy, the advantages corresponding to each token in the response part (this is fixed and does not change with the real-time update of the actor)
        mask: In the old strategy, the mask situation corresponding to the response part is fixed and does not change with the real-time update of the actor)
        
        The reason why logprobs is introduced to calculate actor_loss is because we do not want the policy to be updated too much each time to prevent model training from being distorted.
        
        self.cliprange: default value is 0.2
        """
        ## policy gradient loss
        # ------------------------------------------------------------------------------------------------
        # Calculate the KL divergence between the old and new strategies
        # ------------------------------------------------------------------------------------------------
        log_ratio = (logprobs - old_logprobs) * mask
        ratio = torch.exp(log_ratio)
        # ------------------------------------------------------------------------------------------------
        # Calculate original loss and truncated loss
        # ------------------------------------------------------------------------------------------------
        pg_loss1 = -advantages * ratio
        pg_loss2 = -advantages * torch.clamp(ratio, 1.0 - self.cliprange, 1.0 + self.cliprange)
        pg_loss = torch.sum(torch.max(pg_loss1, pg_loss2) * mask) / mask.sum() # Finally, the average loss of each non-mask response token is taken as the final loss
        return pg_loss

```
</details>

### Construct Critic Loss

Notice that in the construction of advantage, we also get `returns`, which are regarded as the actual returns of each token.


$$
Ret_t = \left( R_t + \gamma \cdot V_{t+1} - V_t \right) + \gamma \cdot \lambda \cdot Adv_{t+1} + V_t = R_t + \gamma \cdot \left( V_{t+1} + \lambda \cdot Adv_{t+1} \right)
$$

The estimated return is $V_t$, and then we construct MSE loss to minimize the difference between the estimated return and the actual return.

$$
critic \_ loss = \left( Ret_t - V_t \right)^2
$$

It seems that $Ret_t - V_t$ is $Adv_t$,** but the `values` actually used is the output of the value model in multiple rounds of updates, that is, the new value, and `returns` is the actual return (old returns) fixed at the beginning of multiple rounds of updates, so $Ret_t - V_t$ is not $Adv_t$. **

The pseudo code is as follows:

<details>
<summary>Pseudocode</summary>

```python
def critic_loss_fn(self, values, old_values, returns, mask):
        """
        values: The estimated expected return generated by the real-time critic (it is variable and changes with the iteration of ppo epoch)
        old_values: The estimated expected return (a fixed value) run by old critics
        returns: actual expected return
        mask: mask of the response part
        
        self.cliprange_value = 0.2
        """
        ## value loss
        # Use the old value to constrain the new value
        values_clipped = torch.clamp(
            values,
            old_values - self.cliprange_value,
            old_values + self.cliprange_value,
        )
        if self.compute_fp32_loss:
            values = values.float()
            values_clipped = values_clipped.float()
        
        # The loss of the critic model is defined as (estimated expected return - actual expected return) **2
        vf_loss1 = (values - returns)**2
        vf_loss2 = (values_clipped - returns)**2
        vf_loss = 0.5 * torch.sum(
            torch.max(vf_loss1, vf_loss2) * mask) / mask.sum() # Similarly, in the end, the critic loss is averaged to each token.
        return vf_loss
```
</details>

### Update process

1. Prepare a batch of `prompts`;
2. Input the `prompts` of this batch to the Actor, and decode them to obtain `responses`;
3. Input `prompt + responses` to Critic/Reward/Reference, and calculate the values of all tokens, the reward of the last token and the log probs of all tokens respectively. According to the terminology of reinforcement learning, these data are called experiences;
4. Calculate actor loss and critic loss in multiple rounds based on experiences and update the Actor and Critic models.

For step 4, of course we can update actor and critic once for each round of experiences, but in order to maximize the use of this batch of experiences, we update actor and critic multiple rounds. We call the log probs and values ​​before the start of multiple rounds of updates in experiences as old log probs and old values ​​(reward will not be calculated in multiple rounds). In each round, actor and critic will generate new log probs and new values, then calculate actor loss and critic loss based on old, and then update the parameters.
